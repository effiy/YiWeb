### 主要价值

- 🧪 架构验证自动化 — 模块存在性、依赖完整性、信任边界可自动检查
- 📋 文档覆盖审计 — 确保每个模块都有对应的基线文档
- 🔒 安全合规回归 — 每次变更后验证安全约束未被破坏
- 🚦 Gate A 交接信号 — 为所有后续 stories 提供架构健康度检查

## 正常用例

### TC-N1: 模块存在性验证

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 关联 | FP7 |
| Given | YiWeb 项目源码完整可访问 |
| When | 运行模块存在性检查：验证技术评审.md §8 模块职责矩阵中的每个模块在文件系统中存在 |
| Then | 14 个模块全部存在，模块入口文件可访问，目录结构匹配 |
| 验证命令 | `ls src/views/aicr/ src/views/claude/ src/views/story/ src/core/services/ cdn/components/ cdn/utils/` |

### TC-N2: 依赖完整性验证

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 关联 | FP8 |
| Given | 技术评审.md §8 依赖矩阵已定义全部模块间依赖关系 |
| When | 验证每个依赖关系：被依赖方的入口文件存在，依赖方的 import 语句正确 |
| Then | 全部依赖关系有效，import 路径可解析 |
| 验证命令 | `grep -r "from '/src/core/services/" src/views/` |

### TC-N3: 信任边界验证

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 关联 | FP4 |
| Given | 技术评审.md §4 定义了认证架构和信任边界 |
| When | 验证：所有 fetch 调用显式设置 credentials:'omit'，所有请求携带 X-Token 头 |
| Then | 无遗漏的 credentials 设置，Token 管理路径完整 |
| 验证命令 | `grep -r "credentials" src/core/services/` + `grep -r "X-Token" src/core/services/` |

### TC-N4: 文档覆盖验证

| 字段 | 内容 |
|------|------|
| 优先级 | P1 |
| 关联 | FP7 |
| Given | 模块职责矩阵中定义了全部已知模块 |
| When | 检查每个模块在 docs/故事任务面板/ 下是否有对应的故事目录 |
| Then | 全部 5 个基线模块（aicr/claude/story/services/cdn）有故事目录且含 ≥5 基线文档 |
| 验证命令 | `ls docs/故事任务面板/aicr/ docs/故事任务面板/claude/ docs/故事任务面板/story/ docs/故事任务面板/services/ docs/故事任务面板/cdn/` |

## 边界用例

### TC-B1: 空模块目录

| 字段 | 内容 |
|------|------|
| 优先级 | P1 |
| 关联 | FP7 |
| Given | 模块目录存在但为空（无入口文件） |
| When | 运行模块存在性检查 |
| Then | 检查报告标注该模块为 "空目录" 而非漏检，建议补充或移除 |

### TC-B2: 新增未注册模块

| 字段 | 内容 |
|------|------|
| 优先级 | P1 |
| 关联 | FP7, FP8 |
| Given | 开发者新增了一个模块目录但未更新模块职责矩阵 |
| When | 运行模块存在性检查（目录扫描 vs 文档条目对比） |
| Then | 检查报告列出 "新增未注册模块" 清单，提示走 /rui update 更新文档 |

### TC-B3: 异常模块名

| 字段 | 内容 |
|------|------|
| 优先级 | P2 |
| 关联 | FP7 |
| Given | 模块目录名不符合命名规范（如含大写字母、特殊字符） |
| When | 运行模块命名规范检查 |
| Then | 检查报告列出不符合规范的模块名，建议修正 |

## 异常用例

### TC-E1: import 链断裂

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 关联 | FP8 |
| Given | 某模块的 import 语句引用了不存在的文件 |
| When | 运行依赖完整性检查 |
| Then | 检查报告标注断裂的 import 链，列出引用方和目标路径 |

### TC-E2: 循环依赖检测

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 关联 | FP8 |
| Given | 模块 A 依赖模块 B，模块 B 依赖模块 A |
| When | 运行依赖拓扑排序检查 |
| Then | 检查报告标注循环依赖环，列出参与模块 |

### TC-E3: 安全约束降级

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 关联 | FP4 |
| Given | 新增代码使用 fetch 但未设置 credentials:'omit' 或未携带 X-Token |
| When | 运行安全约束检查 |
| Then | 检查报告列出违反安全约束的代码位置 |

## 回归用例

### TC-R1: 架构文档同步

| 字段 | 内容 |
|------|------|
| 优先级 | P1 |
| 关联 | AC5 |
| Given | 项目架构发生了变更（如新增视图、修改依赖关系） |
| When | 运行架构文档同步检查：对比技术评审.md 变更记录日期 vs 最近一次架构变更的 git commit 日期 |
| Then | 若文档变更日期晚于代码变更日期 = 通过；否则告警提示更新文档 |

### TC-R2: 跨文档一致性

| 字段 | 内容 |
|------|------|
| 优先级 | P1 |
| 关联 | AC5 |
| Given | 多个文档引用了相同的架构事实（如模块列表） |
| When | 运行跨文档一致性检查：对比故事任务.md、技术评审.md、CLAUDE.md 中的模块清单 |
| Then | 三处引用一致 = 通过；不一致 = 报告差异清单 |
