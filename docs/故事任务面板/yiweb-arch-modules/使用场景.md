### 主要价值

- 🗺️ 快速定位 — 按关键词找到目标模块的入口文件
- 🔗 影响评估 — 修改模块前查看全部下游消费者
- 📋 模块分类索引 — 按类型（通用组件/业务组件/服务/基础设施/插件）浏览
- 🧭 新人导航 — 从模块地图建立系统全貌认知

<a id="sec1"></a>

## §1 角色

| 角色 | 职责 | 关注点 |
|------|------|--------|
| 功能开发者 | 找到要修改的模块位置 | 模块入口路径、文件位置 |
| 影响评估者 | 评估变更的影响范围 | 模块的下游消费者列表 |
| 代码审查者 | 检查新增模块是否符合规范 | 模块的依赖方向和内部依赖 |
| 新加入的开发者 | 理解系统由哪些模块组成 | 模块分类索引和整体结构 |

## §2 场景

### 场景 1: 修改前查下游 — 评估变更影响

- **角色**: 需要修改公共模块的功能开发者
- **前置**: 已确定要修改的目标模块
- **操作流**:

```mermaid
flowchart TD
    A[确定要修改的模块] --> B[在模块地图中查找该模块]
    B --> C{模块在地图中?}
    C -->|是| D[查看下游消费者列表]
    C -->|否| E[全局搜索模块名<br/>补充到模块地图]
    E --> D
    D --> F{下游数量?}
    F -->|0| G[无下游影响<br/>可直接修改]
    F -->|1-3| H[逐下游检查使用方式<br/>修改后逐下游验证]
    F -->|>3| I[按影响级别分组<br/>先改模块再逐组验证]
```

- **后置**: 明确变更影响范围，知道需要验证哪些下游
- **异常**: 模块在地图中但下游列表不完整 → 全局搜索 import 语句补充

| 步骤 | 操作 | 参考 |
|------|------|------|
| 1 | 在模块地图中定位目标模块 | 模块地图总表 |
| 2 | 读取该模块的下游消费者列 | 模块三元组的第三元 |
| 3 | 按下游列表逐项检查使用方式 | 下游模块的入口文件 |

### 场景 2: 按分类浏览 — 找到特定类型的模块

- **角色**: 需要找某个通用组件或工具的新人
- **前置**: 知道要找的模块类型但不知道具体文件路径
- **操作流**:

```mermaid
flowchart TD
    A[需要找一个模块] --> B{知道属于哪种类型?}
    B -->|通用界面组件| C[浏览通用组件分类<br/>YiModal/YiButton/YiTag/...]
    B -->|渲染相关| D[浏览渲染插件分类<br/>SanitizePlugin/MermaidPlugin/...]
    B -->|接口请求| E[浏览服务层分类<br/>crud/requestHelper/...]
    B -->|底层工具| F[浏览基础设施分类<br/>log/error/storage/...]
    B -->|不知道| G[遍历四大分类<br/>逐一排除]
    C --> H[找到目标模块<br/>获取入口路径]
    D --> H
    E --> H
    F --> H
    G --> H
```

- **后置**: 找到目标模块的入口文件路径
- **异常**: 四大分类均未找到 → 可能是新模块需求，按项目规范新建

| 步骤 | 操作 | 参考 |
|------|------|------|
| 1 | 确定模块所属类型 | 五大分类定义 |
| 2 | 浏览该分类下的全部模块 | 模块地图对应分类表 |
| 3 | 匹配模块名称和职责描述 | 模块的职责列 |

### 场景 3: 依赖方向校验 — 审查代码合规性

- **角色**: 代码审查者
- **前置**: 有新增或修改的模块代码
- **操作流**:

```mermaid
flowchart TD
    A[收到代码变更] --> B[识别新增或修改的 import 语句]
    B --> C[确定被 import 模块的类型]
    C --> D[确定当前模块的类型]
    D --> E{依赖方向符合规范?}
    E -->|组件引用组件 → 允许| F[通过]
    E -->|视图引用组件 → 允许| F
    E -->|视图引用服务 → 允许| F
    E -->|组件引用视图 → 违规| G[标记违规]
    E -->|服务引用视图 → 违规| G
    F --> H[记录审查结果]
    G --> H
```

- **后置**: 所有跨模块引用通过方向校验
- **异常**: 发现组件引用视图 → 标记为架构违规，要求调整

| 步骤 | 操作 | 参考 |
|------|------|------|
| 1 | 列出本次变更的全部新增 import | 代码变更 diff |
| 2 | 确定每个 import 的来源模块类型 | 模块地图的分类列 |
| 3 | 检查 cdn/ 模块是否 import 了 src/ 模块 | 依赖方向约束 |

### 场景 4: 全局搜索补充下游 — 完善模块地图

- **角色**: 维护模块地图的架构决策者
- **前置**: 怀疑某些模块的下游列表不完整
- **操作流**:

```mermaid
flowchart TD
    A[选取目标模块] --> B[提取模块入口文件名]
    B --> C[全局搜索该文件名的 import 语句]
    C --> D[排除 node_modules 和 .git 结果]
    D --> E[对比搜索结果与地图下游列表]
    E --> F{有遗漏?}
    F -->|是| G[补充遗漏的下游模块]
    F -->|否| H[下游列表完整]
    G --> I[更新模块地图]
    H --> I
```

- **后置**: 目标模块的下游列表与全局搜索结果一致
- **异常**: 搜索结果包含不属于本仓库的路径 → 标记为外部消费者

| 步骤 | 操作 | 参考 |
|------|------|------|
| 1 | 提取模块入口文件的相对路径 | 模块地图的入口列 |
| 2 | 全局搜索该路径的引用 | grep -r "路径" --include="*.js" |
| 3 | 对比并补充遗漏项 | 模块地图的下游消费者列 |
