### 2.2 六维评估

| 维度 | 评级 | 证据 | 说明 |
|------|:--:|------|------|
| 耦合 | B | 筛选逻辑在 index.js 中内联 | buildFilterContext 与组件耦合，可抽取为独立 hook |
| 稳定性 | A | computed 响应式缓存，无副作用 | 筛选纯计算，输入→输出确定 |
| 扩展性 | B | 新增标签维度需修改 buildFilterContext | 标签维度硬编码为三级，可抽象为通用管道 |
| 可测试性 | C | 无测试框架 | 纯函数可手动验证，但无自动化回归 |
| 安全边界 | A | 输入校验 + XSS 防护 + 认证链路 | 安全审计 STRIDE 全覆盖 |
| 依赖方向 | A | 单向：store → computed → render | 符合 DAG，无循环依赖 |

## §4 经验沉淀

| # | 经验 | 类别 | 来源阶段 | 适用范围 |
|---|------|------|---------|---------|
| 1 | 文件命名约定不可靠，结构化数据（文件夹名）优于字符串提取（getSuffix） | 设计原则 | code | 所有依赖文件路径/名称做判断的场景 |
| 2 | 标签分类需结合语义（storyNames）而非仅靠深度判断 | 设计原则 | code | 文件树筛选、标签系统设计 |
| 3 | 跨视图交互模式尽早统一可避免后期三倍对齐成本 | 工程实践 | code | 多视图项目中共享 UI 模式的开发 |
| 4 | 文档基线应在 code 阶段强制生成实施报告，而非事后补充 | 流程改进 | doc → code | 所有 rui code 阶段的交付物检查 |
| 5 | 纯前端项目的筛选计算用 computed 缓存即可满足性能，无需引入虚拟滚动 | 性能认知 | verify | 文件量 < 500 的前端项目 |
