## §1 使用场景

| 维度 | 内容 |
|------|------|
| **角色** | 功能开发者 |
| **前置** | 有实际的文件变更，当前在规范命名的功能分支上 |
| **操作流** | 提交触发 → 找出本次变更了哪些文件 → 有文件变更? → 按文件类型分类(文档/代码/配置) → 变更了文档?(检查文档格式+引用) / 变更了代码?(检查安全+日志+架构) / 变更了配置?(检查环境切换) → 全部通过(允许提交) / 有失败(阻断提交列出失败项) |
| **后置** | 增量自检全部通过则提交成功；有失败则提交被拒绝 |
| **异常** | 变更中包含非文本文件 → 自动跳过，不阻塞 |

## §2 技术评审

| 评审项 | 结论 | 说明 |
|--------|------|------|
| 增量检测方法 | 通过 | git diff 识别变更文件 |
| 按类型分类检查 | 通过 | 文档/代码/配置 三类分别检查 |
| 性能 | 通过 | 仅检查变更文件，远快于全量 |

### 增量检查分类

| 变更类型 | 触发检查 |
|---------|---------|
| 文档 (.md) | 检查文档格式 + 检查文档引用 |
| 代码 (.js) | 检查安全(嵌入密码?) + 检查日志(规范工具?) + 检查架构(跨层引用?) |
| 配置 (.json/.yaml) | 检查配置(环境切换正确?) |
