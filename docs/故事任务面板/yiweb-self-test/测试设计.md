## 概述

本文档定义 YiWeb 自主测试方案自身的测试策略。自检方案作为一种"元测试"，其正确性直接影响 rui 管道的可靠性。每个检查项都需要经过正向通过、反向失败、边界抗扰三类测试，确保不漏报（false negative）也不误报（false positive）。

本测试设计覆盖 6 大类共 18 个测试用例，按检查类别组织。

---

### 主要价值

- ✅ 确保每条 P0 检查规则在正反两方向上都正确运行 —— 合规代码通过，违规代码被捕获
- 🛡️ 确保自检脚本自身的边界健壮性 —— 在文件不存在、编码异常、权限不足等异常条件下不崩溃
- 🔄 为 tester 提供可重复执行的标准化测试用例，支持手动和自动化两种执行方式
- 📊 测试结果直接关联到 Gate 判定逻辑，确保管道阻断决策的可信度

---

## §1 测试环境与约定

| 维度 | 说明 |
|------|------|
| 测试框架 | Node.js 内置 `node:test` (v18+) 或手动逐条执行 |
| 测试数据 | 在 `__fixtures__/` 目录下维护"合规项目"和"违规项目"两份 fixture |
| 执行方式 | `node self-test.mjs --mode=test` 进入测试模式，逐条执行并输出 TAP 格式结果 |
| 通过标准 | 全部 P0 相关用例通过；P1 相关用例允许 ≤1 个失败（记录为已知问题） |

## §2 测试用例

### 2.1 CLAUDE.md 完整性 (ID: claude-completeness)

| 编号 | 用例名称 | 前置条件 | 输入 | 预期结果 |
|------|---------|---------|------|---------|
| TC-CLD-01 | 完整的 CLAUDE.md 通过检查 | CLAUDE.md 包含全部 5 个标记段（项目画像、项目约束、执行准则、退化对策、安全面） | 标准 CLAUDE.md | status: pass |
| TC-CLD-02 | 缺少"退化对策"段被检出 | CLAUDE.md 有 4 个标记段但缺少"## 退化对策" | 残缺 CLAUDE.md | status: fail, 详细说明缺少的段名 |
| TC-CLD-03 | CLAUDE.md 文件不存在 | 项目根目录无 CLAUDE.md | 不存在的文件路径 | status: fail, 明确提示"CLAUDE.md 不存在"而非报内部错误 |
| TC-CLD-04 | CLAUDE.md 为空文件 | CLAUDE.md 存在但大小为 0 | 空文件 | status: fail, 提示"文件为空" |

### 2.2 README.md 领域语言 (ID: readme-domain)

| 编号 | 用例名称 | 前置条件 | 输入 | 预期结果 |
|------|---------|---------|------|---------|
| TC-RDM-01 | 包含 ≥3 个领域术语的 README 通过 | README.md 中包含"零构建链"、"视图隔离"、"ESM 原生"等术语及其定义 | 标准 README.md | status: pass, 列出检测到的术语数 |
| TC-RDM-02 | 仅有 1 个术语被检出 | README.md 中只有一个术语定义 | 术语不足的 README.md | status: fail, 提示"仅检测到 1 个领域术语，需要至少 3 个" |
| TC-RDM-03 | 术语出现但未定义 | README.md 提及术语但未给出解释 | 有提及无定义的 README.md | status: fail, 提示"术语 xxx 出现但未定义" |
| TC-RDM-04 | README.md 文件不存在 | 项目根目录无 README.md | 不存在的文件路径 | status: fail, 明确提示"README.md 不存在" |

### 2.3 故事目录结构 (ID: story-structure)

| TC-STY-02 | 某目录缺少"安全审计.md" | 某个故事目录仅包含 4 个文档，缺少安全审计 | 残缺目录 | status: fail, 列出缺失项: "安全审计.md" |

### 2.4 分支隔离 (ID: branch-isolation)

| 编号 | 用例名称 | 前置条件 | 输入 | 预期结果 |
|------|---------|---------|------|---------|
| TC-BRH-01 | 功能分支通过检查 | 当前在 `feat/yiweb-self-test` 分支 | `git branch --show-current` 输出 `feat/yiweb-self-test` | status: pass |
| TC-BRH-02 | main 分支被检出 | 当前在 `main` 分支 | `git branch --show-current` 输出 `main` | status: fail, 提示"禁止在 main 分支执行写操作" |
| TC-BRH-03 | 非规范分支名 | 分支名为 `feature/xxx`（非 `feat/` 前缀） | `git branch --show-current` 输出 `feature/xxx` | status: fail, 提示"分支名应为 feat/<name> 格式" |
| TC-BRH-04 | git 不可用 | 不在 git 仓库中 | `git branch` 返回错误 | status: fail, 降级模式标记，明确说明"git 不可用" |

### 2.5 版本一致性 (ID: version-consistency)

| 编号 | 用例名称 | 前置条件 | 输入 | 预期结果 |
|------|---------|---------|------|---------|
| TC-VER-01 | 版本号全部一致 | CLAUDE.md v1.0.0, 文档 F.meta v1.0.0, 代码注释中的版本标记均为 1.0.0 | 版本一致的项目 | status: pass |
| TC-VER-02 | CLAUDE.md 与文档版本不一致 | CLAUDE.md 标 v1.1.0，但文档 F.meta 标 v1.0.0 | 版本冲突 | status: fail, 列出不一致的文件和各自版本 |
| TC-VER-03 | 缺少版本号标记 | 某文档 F.meta 标头无版本号字段 | 格式残缺的文档 | status: fail, 提示"缺少版本号标记" |
| TC-VER-04 | 语义版本格式错误 | 版本号写成 `1.0`（缺少 patch）或 `v1.0.0-beta` | 格式错误 | status: fail, 提示"版本号应为 x.y.z 语义版本格式" |

### 2.6 安全合规 (ID: security-no-secrets / security-fetch-credentials / security-no-raw-log)

| 编号 | 用例名称 | 前置条件 | 输入 | 预期结果 |
|------|---------|---------|------|---------|
| TC-SEC-01 | 无硬编码密钥 | 代码中无 `password = "xxx"` / `token: "xxx"` / `secret: "xxx"` 模式 | 清洁代码 | status: pass |
| TC-SEC-02 | 检测到硬编码密码 | 代码中包含 `const password = "admin123"` | 违规代码 | status: fail, 列出文件名+行号+违规片段 |
| TC-SEC-03 | 检测到明文 Token | 代码中包含 `token: "sk-abc123def456"` | 违规代码 | status: fail, 列出文件名+行号 |
| TC-SEC-04 | 合法变量名不误报 | 代码中有 `passwordInput`、`tokenElement` 等引用 DOM 元素的变量名 | 合法代码 | status: pass, 不误报 |
| TC-SEC-05 | 所有 fetch 有 credentials 设置 | 所有 `fetch(` 调用后紧跟或包含 `credentials: 'omit'` | 合规代码 | status: pass |
| TC-SEC-06 | fetch 缺少 credentials | 某处 `fetch(url)` 未设置 credentials | 违规代码 | status: fail, 列出位置 |
| TC-SEC-07 | 无裸 console.log | 代码使用 `logInfo/logWarn/logError` | 合规代码 | status: pass |
| TC-SEC-08 | 检测到 console.log | 代码中直接调用 `console.log("debug")` | 违规代码 | status: fail (P1), 列出位置 |

## §5 边界测试矩阵

| 编号 | 用例名称 | 边界条件 | 预期行为 |
|------|---------|---------|---------|
| TC-BND-01 | 超大文件 | 单个 JS 文件 > 1MB | 正常处理（不报内存错误），耗时记录在报告中 |
| TC-BND-02 | 二进制文件 | 变更中包含 .png / .ico 等二进制文件 | 自动跳过，不检查二进制文件 |
| TC-BND-03 | 编码异常 | 文件为 GBK 编码而非 UTF-8 | 报 warning 而非 crash，跳过该文件内容检查 |
| TC-BND-04 | 符号链接 | 项目中有 symlink 指向外部路径 | 跟随 symlink 检查实际文件内容 |
| TC-BND-05 | 权限不足 | 脚本对某文件无读权限 | 报 warning，skip 该文件，不阻断 |
| TC-BND-06 | 并发写入 | 自检过程中有其他进程修改文件 | 自检基于读取时的快照，不保证写入一致性；报告中标注"快照时刻" |

## §6 端到端场景测试

| 编号 | 场景 | 测试步骤 | 预期结果 |
|------|------|---------|---------|
| TC-E2E-01 | rui init 后全量自检 | 1. 准备合规项目 fixture 2. 执行 `selfTest.run('full')` 3. 检查输出报告 | 全部 15 项 pass，耗时 < 2s，退出码 0 |
| TC-E2E-02 | 违规项目的全量自检 | 1. 准备违规项目 fixture（缺少 CLAUDE.md + 有硬编码密钥） 2. 执行 `selfTest.run('full')` | 至少 2 项 fail，退出码非 0，报告中含修复指引 |
| TC-E2E-03 | pre-commit 增量自检 | 1. 修改 3 个文件（1 合规 + 1 违规 + 1 二进制） 2. `git add` 3. 执行 `selfTest.run('incremental')` | 仅检查 3 个变更文件，合规文件 pass，违规文件 fail，二进制 skip |
| TC-E2E-04 | 降级模式自检 | 1. 临时移除 git 2. 执行 `selfTest.run('degraded')` | branch-isolation 标记为 skipped，其他项正常执行 |
