## 概述

本文档对 YiWeb 自主测试方案自身进行安全审计。自检系统作为 rui 管道的守门人，其自身的安全性直接决定了整个开发流程的安全基线 —— 如果自检可以被绕过，那么检查就失去了意义。因此，除了检查别人，自检系统必须先经得起检查。

审计采用 STRIDE 威胁模型，覆盖自检系统的规则配置、执行引擎、报告输出、历史存储、以及与 rui 管道的集成接口。

---

### 主要价值

- 🔒 防止自检被恶意绕过或伪造 —— 确保检查结果的可信度
- 🛡️ 识别自检脚本自身的攻击面 —— 配置文件篡改、内存泄漏、信息泄露
- 📋 建立自检系统的合规基线 —— 六项合规检查确保自检系统本身符合安全最佳实践
- 🔄 降级路径不得成为攻击入口 —— 紧急跳过开关必须有使用审计

## §1 自检系统攻击面

### 1.1 系统边界

```
┌──────────────────────────────────────────────────────────┐
│                     自检系统边界                            │
│                                                           │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐  │
│  │ 规则配置文件   │   │  执行引擎     │   │  报告输出     │  │
│  │ (JSON)       │──→│  (JS)        │──→│  (JSONL)     │  │
│  └──────────────┘   └──────┬───────┘   └──────────────┘  │
│                            │                              │
│                   ┌────────▼───────┐                      │
│                   │  文件系统访问    │                      │
│                   │  (fs.read)     │                      │
│                   └────────┬───────┘                      │
│                            │                              │
└────────────────────────────┼──────────────────────────────┘
                             │
                    ┌────────▼───────┐
                    │  项目源文件     │
                    │  (被检查对象)   │
                    └────────────────┘
```

### 1.2 攻击面清单

| 攻击面 | 暴露点 | 风险等级 |
|--------|--------|---------|
| 规则配置篡改 | `self-test-rules.json` 可被本地修改 | 高 |
| 报告伪造 | 历史报告可被直接编辑 | 中 |
| 环境变量滥用 | `YIWEB_SKIP_SELF_TEST` 可被设置从而跳过全部检查 | 高 |
| 检查函数注入 | 动态调用 `checkFn` 如果配置被篡改可执行任意代码 | 高 |
| 文件读取路径穿越 | 如果检查规则中的路径未正确校验，可能遍历到项目外文件 | 中 |
| 报告信息泄露 | 检查结果中包含文件路径和代码片段，可能泄露项目结构 | 低 |
| 降级路径滥用 | 利用降级模式跳过关键检查项 | 中 |

## §2 STRIDE 威胁模型

### 2.1 欺骗 (Spoofing)

| 威胁 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| S-01 | 攻击者伪造自检报告，将失败状态改为成功 | 管道在破损基线上继续运行 | 报告文件附加 HMAC 签名，Gate 校验签名；签名密钥由 rui 管理 |
| S-02 | 攻击者替换自检脚本为假的"全通过"脚本 | 完全绕过检查 | 脚本 hash 值硬编码在 rui 管道配置中，启动前校验 |
| S-03 | 攻击者修改 git 分支名（通过 `git branch -m`）绕过 branch-isolation | 分支检查失效 | branch-isolation 检查同时验证分支的 parent commit 是否在 main 上 |

### 2.2 篡改 (Tampering)

| 威胁 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| T-01 | 修改 `self-test-rules.json`，删除 P0 检查项或降低其优先级 | 关键检查项失效 | 规则文件 hash 纳入自检校验范围（Meta-check: 自检规则自身的 hash 检查） |
| T-02 | 修改历史报告文件，隐藏过去的失败记录 | 审计追溯断裂 | 历史报告采取 append-only 模式，一旦写入不可修改；异常修改可通过文件时间戳检测 |
| T-03 | 修改 fixture 项目文件使测试用例误通过 | 自检测试失效 | fixture 文件的 hash 快照作为测试前置验证 |

### 2.3 否认 (Repudiation)

| 威胁 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| R-01 | 开发者设置 `YIWEB_SKIP_SELF_TEST=1` 后声称不知情 | 推卸责任 | 环境变量设置行为记录到终端日志 + 自检报告中标注 "SKIPPED BY ENV VAR" |
| R-02 | 自检执行后无人承认修改过配置导致规则失效 | 事后追责困难 | 规则配置属于 git 版本管理，篡改有 commit 记录 |

### 2.4 信息泄露 (Information Disclosure)

| 威胁 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| I-01 | 自检报告揭示了项目的完整目录结构 | 攻击者了解项目布局 | 报告不应提交到公开仓库；存储在 `.memory/` 目录中，通过 `.gitignore` 排除 |
| I-02 | 安全合规检查输出中包含了疑似密钥的代码片段 | 报告中泄漏了实际的密钥 | 输出中对疑似密钥进行脱敏处理（显示前 2 位 + 后 2 位，中间用 `***` 替换） |
| I-03 | 终端输出中的文件路径暴露了本地开发环境路径 | 暴露开发者用户名或目录结构 | 终端输出使用相对路径（相对于项目根目录），不输出绝对路径 |

### 2.5 拒绝服务 (Denial of Service)

| 威胁 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| D-01 | 恶意构造超大文件（>100MB）导致自检内存溢出 | 自检崩溃，管道中断 | 单个文件读取上限 10MB；超过上限标记为 skipped |
| D-02 | 大量小文件（10000+）导致全量扫描超时 | 自检长时间无响应 | 全量扫描限制文件数上限 5000；超出部分标记为 skipped |
| D-03 | 检查规则执行死循环 | 自检永不结束 | 单检查项超时 500ms；全局超时 10s |

### 2.6 权限提升 (Elevation of Privilege)

| 威胁 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| E-01 | 自检脚本以 root 运行，恶意规则配置通过动态 eval 获取系统权限 | 攻击者可执行任意系统命令 | 规则配置不支持 `eval`；checkFn 必须是预注册函数名，不支持任意 JS 代码 |
| E-02 | 攻击者通过路径穿越读取 `/etc/passwd` 等系统文件 | 系统信息泄露 | 所有文件读取路径限制在项目根目录内（`path.resolve` + `startsWith` 校验） |

## §3 降级路径安全

### 3.1 降级模式风险

降级模式（degraded）的设计初衷是在某些工具不可用时仍能执行部分检查。但这同时打开了潜在的攻击窗口。

| 风险 | 攻击场景 | 缓解措施 |
|------|---------|---------|
| 故意制造工具不可用 | 攻击者临时卸载 git，触发降级模式，绕过 branch-isolation 检查 | 降级模式触发时记录完整的环境快照（PATH、工具版本），供事后审计 |
| 降级后跳过关键检查 | 攻击者编写恶意脚本伪装工具不可用，使检查项被标记为 skipped | 工具可用性检测使用白名单方式：必须确认工具确实不存在才跳过，而非"执行失败即跳过" |

### 3.2 紧急跳过开关

`YIWEB_SKIP_SELF_TEST=1` 作为紧急开关存在以下约束：

| 约束 | 说明 |
|------|------|
| 使用记录 | 每次使用紧急开关都会记录时间戳、执行上下文（管道步骤）、操作者信息到独立审计日志 |
| 时效限制 | 紧急开关在 set 后 30 分钟自动失效，防止忘记取消 |
| 感知提示 | 终端输出红色加粗警告："自检已被跳过！请尽快修复问题并恢复自检" |
| 不可静默 | 紧急开关被设置时，终端输出 MUST 包含醒目的警告消息，不依赖日志级别 |

## §4 合规检查

### 4.1 检查清单

| 编号 | 检查项 | 要求 | 通过条件 |
|------|--------|------|---------|
| CC-01 | 最小权限 | 自检脚本仅需读取文件系统权限，不要求写入或执行权限 | 使用 `fs.promises.readFile`，禁止使用 `fs.writeFile` 或 `child_process.exec` |
| CC-02 | 路径安全 | 所有文件路径解析必须限制在项目根目录内 | 代码审查确认所有路径使用 `path.resolve(projectRoot, relativePath)` 后校验 `startsWith(projectRoot)` |
| CC-03 | 输入校验 | 从配置文件读取的规则必须校验类型和范围 | checkFn 必须是预注册表白名单中的函数名；priority 必须是 "P0"/"P1"/"P2" 之一 |
| CC-04 | 安全日志 | 自检脚本自身的日志不可包含敏感信息 | 日志输出中无绝对路径；错误堆栈中隐去完整文件路径 |
| CC-05 | 配置完整性 | 规则配置文件不可被外部不可信来源修改 | 规则文件纳入 git 版本管理；修改必须有 pull request + review |
| CC-06 | 加密敏感配置 | 任何敏感配置（如签名密钥、HMAC 密钥）不可明文存储 | 密钥通过环境变量或独立 keychain 管理，不写入配置文件 |

### 4.2 合规检查通过矩阵

| 检查项 | CC-01 | CC-02 | CC-03 | CC-04 | CC-05 | CC-06 |
|--------|:-----:|:-----:|:-----:|:-----:|:-----:|:-----:|
| 自检执行引擎 | ✅ | ✅ | ✅ | ✅ | ✅ | ⚠️ (见 §6 未决项) |
| 规则配置文件 | — | — | ✅ | — | ✅ | — |
| 报告输出模块 | — | — | — | ✅ | — | — |

## §5 依赖审计

自检系统依赖的外部能力：

| 依赖 | 用途 | 风险 | 缓解 |
|------|------|------|------|
| Node.js `fs` 模块 | 文件读取 | Node.js 版本漏洞 | 要求 Node.js v18 LTS+ |
| Node.js `path` 模块 | 路径处理 | 无（内置模块） | — |
| `child_process.execSync('git ...')` | 分支检测、diff 获取 | git 版本漏洞 | 使用最小参数集；不传递用户输入到 shell |
| `crypto` 模块 | HMAC 签名生成 | 随机数强度 | 使用 `crypto.randomBytes` 而非 `Math.random` |
| 项目文件系统 | 读取被检查的文件 | 文件被其他进程修改 | 快照式读取（一次性读入内存） |

| 编号 | 描述 | 状态 | 理由 |

| UI-03 | 自检脚本运行在 Node.js 环境，但 YiWeb 本身是前端项目 | 已接受 | 自检是开发工具链的一部分，不是浏览器运行时代码，Node.js 依赖是可接受的 |
