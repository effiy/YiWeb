## 概述

YiWeb 自主测试方案（下称"自检方案"）是项目级健康检查策略，用于在 rui 工作流的各个阶段自动验证项目基线是否完整、一致、合规。自检方案不依赖任何后端服务或第三方工具，仅在浏览器环境或 Node.js 环境中执行文件级与结构级校验。它是 rui 管道的**安全网**，确保每一次 `init` / `code` / `review` / `update` 操作都在可靠的基线上运行。

本方案包含两条故事，覆盖两个维度：运行时的管线健康自检，以及静态的文档基线完整性校验。

---

### 主要价值

- 🔍 每次 `rui init` 后全量扫描项目文件的完整性与一致性，防止不同环境中基线漂移
- 🛡️ 每次 `git commit` 前增量检查变更文件，杜绝破损基线进入版本历史
- 📋 定期（或按需）校验所有故事目录是否具备完整的 5 文档基线，确保可追溯性
- ⚡ 最低代价的回归安全网 —— 自检全部在文件级别完成，耗时 < 2s，无外部依赖
- 🔗 将文档、分支、版本、安全面串联为统一的规则网，避免孤立检查导致盲区

## §1 故事

### §1.1 用户操作

自检方案的主要"用户"是 rui 管道调度器（pm Agent）和各阶段 agent（coder / reviewer / tester）。人类开发者通过 rui 命令间接触发自检，不直接调用检查脚本。

| 角色 | 操作 | 触发时机 |
|------|------|---------|
| pm Agent | `rui init` 后执行全量自检 | 每次项目初始化 |
| coder Agent | `git commit` 前执行增量自检 | 每次编码完成 |
| reviewer Agent | 文档→代码一致性自检 | 技术评审前 |
| tester Agent | 安全面回归自检 | 安全相关变更后 |

---

### Story 1: 管线健康自检

**优先级**: P0  
**故事点**: 3  
**触发**: rui 管道各步骤的 Gate A（前置检查）

#### 描述

在 rui 管道的每个关键步骤完成后，自动执行一组文件结构与内容完整性检查。检查项涵盖项目配置、代码结构、安全约束、分支规范、版本一致性。任意一项未通过即阻断当前步骤，需要人工介入修复。

#### 检查维度

| 检查项 | 检查内容 | 阻断级别 |
|--------|---------|---------|
| CLAUDE.md 完整性 | 必须包含"项目画像"、"项目约束"、"执行准则"、"退化对策"、"安全面"五个标记段 | 阻断（P0） |
| README.md 领域语言 | 必须包含至少 3 个与项目领域相关的术语定义（如"零构建链"、"视图隔离"、"ESM 原生"等），且定义出现在文档前部 | 阻断（P0） |
| 分支隔离 | 当前分支必须为 `feat/<name>` 格式，且从 `main` 分出；禁止在 main 上执行写操作 | 阻断（P0） |
| 版本一致性 | CLAUDE.md 版本号、文档版本号、代码注释中的版本标记三者一致 | 阻断（P0） |
| 故事目录结构 | 每个故事目录下必须具备 5 个基线文档：故事任务.md、使用场景.md、技术评审.md、测试设计.md、安全审计.md | 阻断（P0） |
| 架构层约束 | 视图层不得直接引用 CDN 工具库的存储模块（必须通过 services 层中转）；禁止跨视图 import | 阻断（P0） |
| 安全合规 | 代码中不得硬编码密钥/Token；fetch 必须设置 `credentials: 'omit'`；无裸 `console.log` | 阻断（P0） |

#### 执行模式

- **增量模式**（默认）：仅检查 `git diff --cached` 或最近一次提交中变更的文件
- **全量模式**：扫描整个项目所有相关文件，通常仅在 `rui init` 或版本升级后触发
- **按类别模式**：可指定仅检查某一维度（如仅检查安全合规）

---

### Story 2: 文档基线完整性校验

**优先级**: P0  
**故事点**: 2  
**触发**: rui review 步骤的 Gate A、定期任务

#### 描述

对所有故事目录的文档基线进行完整性校验，确保每个目录都具备规定的 5 份文档，且文档之间的交叉引用有效、无过期引用、版本号一致。

#### 验证维度

| 验证项 | 验证内容 | 阻断级别 |
|--------|---------|---------|
| 文档存在性 | 每个故事目录下必须存在故事任务.md、使用场景.md、技术评审.md、测试设计.md、安全审计.md 五个文件 | 阻断（P0） |
| 交叉引用有效性 | 文档中的相对链接指向的文件必须存在；不允许出现 404 引用 | 阻断（P0） |
| 过期引用检测 | 引用的外部文档（如 CLAUDE.md、技术架构.md、模块地图.md）的版本号不得早于文档生成时的版本 | 警告（P1） |
| F.meta 标头规范 | 每个文档必须包含规范的 F.meta 标头（版本号、日期、分支、CLAUDE.md 链接） | 阻断（P0） |
| 变更记录 | 每个文档底部必须有格式完整的变更记录表 | 阻断（P0） |
| 导航一致性 | 所有文档的 F.nav 块必须对齐，导航链接不得断开 | 阻断（P0） |

## §2 需求

### FP1: 管线健康自检执行器

| 属性 | 值 |
|------|-----|
| 功能点 | FP1 |
| 描述 | 提供一个可在 rui 管道各步骤调用的自检执行器，支持增量/全量/按类别三种模式 |
| AC | 见 §5 AC1-AC5 |

### FP2: 文档基线完整性校验器

| 属性 | 值 |
|------|-----|
| 功能点 | FP2 |
| 描述 | 提供独立的文档基线完整性校验器，验证所有故事目录的文档结构 |
| AC | 见 §5 AC6-AC10 |

### 业务规则

| 编号 | 规则 |
|------|------|
| BR1 | P0 阻断项必须全部通过方可通过 Gate；P0 项失败时输出明确的修复指引 |
| BR2 | 增量模式下的文件变更检测基于 `git diff --name-only`（缓存区 + 工作区） |
| BR3 | 自检结果写入标准化报告（JSON 格式），保留最近 10 次历史记录 |
| BR4 | 禁用手动跳过自检 —— 除非环境变量 `YIWEB_SKIP_SELF_TEST=1`（仅限紧急情况，记录警告日志） |

### 数据约束

| 编号 | 约束 |
|------|------|
| DC1 | 单个自检报告大小不超过 1MB |
| DC2 | 历史报告保留最近 10 份，超出自动清理 |
| DC3 | 检查规则以 JSON 配置文件形式定义，不硬编码在脚本中 |

## §3 成功标准

| 编号 | 标准 | 度量方式 |
|------|------|---------|
| SC1 | 全量自检在包含 200+ 文件的项目中，执行时间不超过 2 秒 | 计时 |
| SC2 | 自检脚本本身不引入任何外部依赖（npm/brew/pip 等） | 依赖检查 |
| SC3 | 增量自检仅检查实际变更的文件（不扫描全项目） | 日志确认 |
| SC4 | P0 阻断项的误报率为 0（不因规则编写不当而对合规代码报错） | 回归测试 |
| SC5 | 自检结果可被 rui 管道解析并自动生成修复任务 | 集成测试 |

---

## §4 范围边界

| 维度 | 范围内 | 范围外 |
|------|--------|--------|
| 检查对象 | 项目文件群（.md / .js / .html / .css） | 后端服务、数据库、部署环境 |
| 检查粒度 | 文件级结构 + 内容正则匹配 | 代码语义分析、运行时行为 |
| 运行环境 | Node.js（CI）+ 浏览器（开发者工具） | Python / Shell 等其他运行时 |
| 输出格式 | JSON 报告 + 终端彩色输出 | GUI、通知推送、聊天机器人 |
| 集成点 | rui 管道 Gate A/B | 独立 CI/CD 平台（Jenkins / GitHub Actions 等） |
| 被检查的故事目录 | 仅 `docs/故事任务面板/` 下的故事子目录 | 其他文档目录（如 `docs/api/`） |

## §5 验收条件 (AC)

### 管线健康自检 (Story 1)

| 编号 | AC |
|------|-----|
| AC1 | 在 `rui init` 后运行全量自检，覆盖全部 7 个检查维度，输出通过/失败明细 |
| AC2 | 全量自检任一 P0 项失败时返回非零退出码，并输出定位到具体文件+行号的修复指引 |
| AC3 | 在 `git commit` 前（pre-commit hook 或手动调用）运行增量自检，仅检查变更文件 |
| AC4 | 增量自检耗时不超过 500ms（基于 ≤20 个变更文件的场景） |
| AC5 | 安全合规检查能检出硬编码的密钥模式（如 `password = "xxx"` 或 `token: "xxx"`） |

### 文档基线完整性校验 (Story 2)

| 编号 | AC |
|------|-----|
| AC6 | 扫描 `docs/故事任务面板/` 下所有子目录，输出哪些目录缺少哪些文档 |
| AC7 | 验证每个文档的 F.meta 标头格式是否完整（版本号、日期、分支、CLAUDE.md 链接四项缺一不可） |
| AC8 | 验证文档中的 Markdown 相对链接指向的文件是否存在 |
| AC9 | 对比文档引用的外部文件的版本号，检测是否有过期引用 |
| AC10 | 输出格式为 JSON 数组，每个目录一个条目，包含目录名、缺失文档列表、过期引用列表 |

## §6 风险与假设

| 编号 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| R1 | 自检脚本本身有 bug 导致误阻断 | 高 —— 阻断整个 rui 管道 | 自检脚本中编写全面的单元测试；提供 `YIWEB_SKIP_SELF_TEST` 紧急开关 |
| R2 | 检查规则配置（JSON）被意外修改 | 中 —— 漏检或误检 | 规则配置文件纳入版本管理；规则文件的 hash 值硬编码在自检脚本中校验 |
| R3 | 项目结构大规模重构导致检查规则大面积失效 | 中 —— 需要重新基线化 | 自检规则配置文件与项目结构解耦；重构时同步更新规则 |
| R4 | 某些检查项（如"跨视图 import"）的正则匹配不够精确 | 低 —— 个别边缘 case 漏网 | 逐一验证已知的合法模式；异常模式默认报警（宁可误报不可漏报） |
| A1 | 项目根目录始终有 CLAUDE.md 作为权威配置来源 | — | — |
| A2 | git 仓库可用且 `git diff` 命令可正常执行 | — | — |
| A3 | 文档编码为 UTF-8，换行符为 LF | — | — |
