# 产物与回写契约

> 本规范统一 `implement-code` 在各阶段生成、读取、回写的文件路径、命名方式和状态字段。  
> **权威原则**：凡涉及路径、文件名、目录结构、状态回写，均以本文件为准。

---

## 1. 文档集路径

`implement-code` 默认处理完整文档目录：

```text
docs/<功能名>/
├── 01_需求文档.md
├── 02_需求任务.md
├── 03_设计文档.md
├── 04_使用文档.md
├── 05_动态检查清单.md
├── 06_实施总结.md
└── 07_项目报告.md
```

其中：

- `05_动态检查清单.md`：阶段 2 / 6 的状态回写目标
- `06_实施总结.md`：任意阻断和最终完成时都必须生成
- `01/02/03/04/07`：阶段 7 的实施状态回写目标
- `07_项目报告.md`：不由 `implement-code` 从零生成；若已存在，必须更新实施状态小节

若无法定位 `<功能名>` 或 `docs/<功能名>/`，仍必须写入兜底运行记录：

```text
docs/99_agent-runs/<YYYYMMDD-HHMMSS>_implement-code.md
```

若 `docs/99_agent-runs/` 不存在，必须先创建目录。该记录必须包含原始用户请求、路径解析结果、阻断原因、缺失文档或目录、建议恢复步骤。

## 2. 测试产物路径

> **P0 规则**：所有由实施流程生成的测试相关文件（spec、原型页、fixtures、mock 数据、快照、截图、下载产物、测试辅助脚本）都必须位于 `tests/` 目录下。禁止在项目根目录、`src/`、`docs/`、`.claude/` 或临时隐藏目录中生成测试文件；如工具默认输出到其他目录，必须移动或重新配置到 `tests/` 后才可进入下一阶段。

### 2.1 Playwright 测试文件

```text
tests/e2e/<功能名>/<场景名>.spec.ts
```

### 2.2 测试原型页面

```text
tests/e2e/<功能名>/pages/<场景名>/index.html
```

### 2.3 共享 mock 数据

```text
tests/e2e/<功能名>/fixtures/<名称>-mock-data.ts
```

### 2.4 截图与下载

```text
tests/screenshots/<功能名>/<场景名>-<状态>.png
tests/downloads/<功能名>/<文件名>
```

### 2.5 测试路径门禁

阶段 1、阶段 2、阶段 6 退出前必须检查测试产物路径：

| 检查项 | 要求 |
|------|------|
| spec 文件 | 只能匹配 `tests/**/*.spec.ts` / `tests/**/*.test.ts` |
| 原型页面 | 只能位于 `tests/e2e/<功能名>/pages/` |
| mock / fixture | 只能位于 `tests/e2e/<功能名>/fixtures/` 或对应 spec 文件内 |
| 截图 / 下载 / 快照 | 只能位于 `tests/screenshots/`、`tests/downloads/`、`tests/snapshots/` |
| 临时测试辅助文件 | 只能位于 `tests/tmp/` 或 `tests/e2e/<功能名>/tmp/`，并在总结中标注是否清理 |

若发现测试文件落在 `tests/` 外，属于 P0 门禁失败：必须停止进入下一阶段，移动到合规路径或删除误产物，并在 `06_实施总结.md` 的“变更文件清单”和“验证门禁结果归档”中记录处置结果。

## 3. 状态回写契约

### 3.1 `05_动态检查清单.md`

| 阶段 | 状态值 | 含义 |
|------|--------|------|
| 阶段 2 | `🏃` | 原型验证通过 |
| 阶段 6 | `✅` | 冒烟测试通过 |
| 阶段 2 / 6 | `❌` | 验证失败 |
| 阶段 2 / 6 | `⚠️` | 需人工确认 |
| 初始 | `⏳` | 未验证 |

### 3.2 回写字段

- 状态列
- 备注列
- 底部“检查总结”的总体进度、待完成项、结论

### 3.3 禁止事项

- 未实际验证就改状态
- 把阶段 2 的 `🏃` 写成最终 `✅`
- 修改检查项优先级绕过门禁
- 正常完成路径中，仍保留未说明原因的 `⏳`、`🏃`、`❌` 或 `⚠️` 应完成项

### 3.4 最终完成门禁

正常完成 `implement-code` 前，必须对 `docs/<功能名>/05_动态检查清单.md` 执行最终完成门禁：

| 检查项 | 完成要求 |
|------|----------|
| P0 检查项 | 全部为 `✅`，且备注包含阶段 6 验证证据、日期或报告引用 |
| P1/P2 应完成项 | 已完成项为 `✅`；延期或需人工确认项必须为 `⚠️` 并写明原因、负责人或后续动作 |
| 阶段 2 临时状态 | 不得仍停留在 `🏃` 作为最终状态；阶段 6 通过后必须升级为 `✅` |
| 待验证状态 | 不得存在未说明原因的 `⏳` |
| 失败状态 | 不得存在 `❌`；如存在则必须按阻断流程处理 |
| 检查总结 | 总体进度、待完成项、结论必须与表格状态一致 |

最终完成门禁结果必须写入 `06_实施总结.md` 的“验证门禁结果归档”和“状态回写记录”。若门禁不通过，禁止执行正常完成通知，必须改用门禁异常 / 阻断通知。

## 4. 功能文档状态回写契约

### 4.1 回写时机

阶段 7 生成 `06_实施总结.md` 后、执行 `import-docs` 前，必须回写同一功能目录下其他文档的状态信息。正常完成和任意阶段阻断都必须执行。

### 4.2 回写文件

| 文件 | 回写要求 |
|------|----------|
| `01_需求文档.md` | 新增或更新 `## 实施状态`，标记需求是否已交付 |
| `02_需求任务.md` | 新增或更新 `## 实施状态`，标记任务完成度和阻断项 |
| `03_设计文档.md` | 新增或更新 `## 实施状态`，标记设计落地情况和偏差 |
| `04_使用文档.md` | 新增或更新 `## 实施状态`，标记用户文档是否可用 |
| `05_动态检查清单.md` | 按本文件第 3 节更新状态列、备注列和检查总结；正常完成时必须通过最终完成门禁 |
| `07_项目报告.md` | 若文件存在，新增或更新 `## 实施状态`，标记报告是否已反映最新代码变更 |
| `06_实施总结.md` | 记录本次状态回写的文件列表、结果、动态检查清单最终完成门禁结果和未回写原因 |

缺失的 P1/P2 文档不阻断交付，但必须在 `06_实施总结.md` 的状态回写记录中说明“文件不存在，已跳过”。缺失 `05_动态检查清单.md` 属于 P0 阻断，应按停止流程处理。

### 4.3 `## 实施状态` 固定结构

除 `05_动态检查清单.md` 外，其他文档统一使用以下小节。若已存在该小节，只更新表格内容，不重复新增。

```markdown
## 实施状态

| 字段 | 状态 |
|------|------|
| 实施状态 | ✅ 已完成 / 🟡 部分完成 / ⛔ 已阻断 |
| 更新时间 | <ISO 时间戳或 YYYY-MM-DD HH:mm> |
| 实施阶段 | 阶段 0 → 7 / 阶段 <N> 阻断 |
| 验证结果 | P0 通过 <N> / 共 <N> 项；P1/P2 摘要 |
| 关联总结 | [06_实施总结.md](./06_实施总结.md) |
| 下一步 | 无需人工介入 / <具体恢复或修复动作> |
```

### 4.4 状态取值

| 场景 | 实施状态 | 下一步 |
|------|----------|--------|
| 阶段 6 冒烟测试全部 P0 通过，阶段 7 总结完成 | `✅ 已完成` | `无需人工介入`，或进入提交 / 评审 / 发布 |
| 部分模块完成但存在 P1/P2 待处理，不阻断主流程 | `🟡 部分完成` | 列出后续优化项 |
| 任意 P0 门禁未通过或必要输入缺失 | `⛔ 已阻断` | 写明恢复起点和需要补齐的内容 |

### 4.5 回写记录

`06_实施总结.md` 必须新增或更新“状态回写记录”小节：

```markdown
## 状态回写记录

| 文件 | 回写结果 | 状态 | 说明 |
|------|----------|------|------|
| `01_需求文档.md` | 已更新 / 跳过 / 失败 | ✅ / 🟡 / ⛔ | <原因> |
```

禁止事项：

- 未完成验证就把状态写成 `✅ 已完成`
- 阻断时只写实施总结，不回写其他功能文档状态
- 在同一文件中重复新增多个 `## 实施状态` 小节
- 因项目报告尚未更新真实变更内容，就把 `07_项目报告.md` 标记为已反映最新代码变更

---

## 5. 实施总结契约

总结文件路径固定为：

```text
docs/<功能名>/06_实施总结.md
```

适用两种场景：

1. 阶段 7 正常完成时生成完整总结
2. 任意阶段阻断时先生成阻断总结，后续若恢复成功，可由完整总结覆盖

---

## 6. 命名约定

- `<功能名>`：与 `docs/<功能名>/` 目录一致
- `<场景名>`：来自 `02_需求任务.md` 的主要操作场景，保持语义稳定
- `<状态>`：使用英文小写，如 `initial`、`after-action`、`error`

禁止：

- 在 `SKILL.md` 或其他 rule 中再单独定义另一套路径
- 混用 `动态检查清单.md` / `05_动态检查清单.md`
- 混用 `实施总结.md` / `06_实施总结.md`
- 因无法定位功能目录而不写入任何 `docs/` 文件
- 在 `tests/` 以外生成或保留任何测试相关文件
