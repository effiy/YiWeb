# 需求任务文档规范

## 文档概述

- 需求任务文档是对需求文档中用户故事的细化和功能分解，用于指导开发实施
- 必须从对应需求文档提取用户故事表格，保持需求与任务的一致性
- 上游文档：需求文档（位于 `docs/01_需求文档/`）
- 下游文档：设计文档

## 文档结构（严格遵循）

### 1. 文档头部（必须包含）

```markdown
# {功能名称}

> **文档版本**: v1.0 | **最后更新**: {日期} | **维护者**: YiAi 团队
>
> **关联文档**: [需求文档](../01_需求文档/{文件名}.md) | [设计文档](../03_设计文档/{文件名}.md) | [使用文档](../04_使用文档/{文件名}.md)
>

[功能概述](#功能概述) | [功能分析](#功能分析) | [功能详情](#功能详情) | [验收标准](#验收标准) | [使用场景示例](#使用场景示例)

---
```

### 2. 功能概述

- 简要说明功能的目标和范围，100-200字
- 列出 3 个核心价值点，使用 🎯⚡📖 图标

### 3. 功能分析（必须包含 Mermaid 图表）

必须包含 4 种 Mermaid 图表：

- **功能分解图**：`graph TB` - 展示功能模块和子功能
- **用户流程图**：`flowchart TD` - 展示用户操作流程
- **功能流程图**：`flowchart TD` - 展示系统功能处理流程
- **完整时序图**：`sequenceDiagram` - 展示组件间交互时序

每个图表下方必须添加明确的说明文字。

### 4. 用户故事表格（必须从需求文档提取）

- 完整提取自需求文档的用户故事表格
- 保持优先级图标（🔴🟡🟢）和格式不变
- 包含过程生成文档和产出智能文档列
- **必须包含主要操作场景描述**：在用户故事中明确列出主要操作场景，用于后续检查清单的动态生成

### 5. 主要操作场景定义（必须包含）

为每个用户故事明确列出主要操作场景，格式如下：

```markdown
#### 🎯 主要操作场景：{场景名称}

**场景描述**：{简要描述这个操作场景}

**前置条件**：{列出执行此操作前需要满足的条件}

**操作步骤**：
1. {步骤1}
2. {步骤2}
3. {步骤3}

**预期结果**：{描述操作完成后的预期结果}

**验证关注点**：{列出验证此场景需要检查的关键点}

**相关设计文档章节**：{指向设计文档中对应的实现章节}
```

主要操作场景的要求：
- 每个 P0 用户故事至少包含 2 个主要操作场景
- 每个 P1 用户故事至少包含 1 个主要操作场景
- 场景应覆盖核心用户路径和关键功能点
- 场景描述应具体、可操作、可验证

### 6. 影响分析（必须包含）

> **强制执行**：生成需求任务文档前，必须对整个项目执行完整的影响分析，使用 Grep 搜索所有引用处，产出引用清单，避免改动遗漏。

#### 执行步骤

1. **确定核心标识符**：从需求文档中提取本次功能涉及的核心模块名、组件名、事件名、Store key、路由路径、CSS 类名等关键标识符，形成"搜索词列表"。
2. **全项目 Grep 搜索**：对每个搜索词使用 Grep 工具在整个仓库执行搜索，记录所有命中路径与行号。
3. **排除无关结果**：排除 `node_modules/`、`dist/`、`*.lock`、文档自身等非业务文件。
4. **分类标注影响级别**：
   - 🔴 **直接修改**：本次功能必须变更该文件/代码行
   - 🟡 **间接影响**：依赖该模块，需在任务中列明兼容性考虑
   - 🟢 **参考引用**：仅作参考，无需修改

#### 引用清单格式

| 搜索词 | 命中文件 | 行号 | 影响级别 | 说明 |
|--------|----------|------|----------|------|
| `{标识符1}` | `src/path/to/file.ts` | L42 | 🔴 直接修改 | {原因} |
| `{标识符2}` | `src/another/file.vue` | L17 | 🟡 间接影响 | {原因} |

> 若某搜索词未命中任何结果，在表格中标注"未找到引用"，不可省略该行。

#### 改动范围汇总

- **需直接修改的文件数**：{N} 个
- **需验证兼容性的文件数**：{N} 个
- **未覆盖风险**：{列举可能遗漏的场景，或写"无"}

---

### 7. 功能详情

- 按功能点分章节
- 每个功能点包含：功能说明、价值、解决的痛点、收益
- 复杂功能点可添加 Mermaid 图表

### 8. 验收标准

- **P0 - 必须通过**：核心功能，不实现无法发布
- **P1 - 应该通过**：重要功能，建议实现
- **P2 - 可以有**：锦上添花，可选实现

### 9. 使用场景示例

- 每个场景包含：背景、操作、结果
- 使用 📋🎨 等图标标识不同场景

## Mermaid 图表规范

- 功能分解用 `graph TB`
- 用户流程用 `flowchart TD`
- 功能流程用 `flowchart TD`
- 时序图用 `sequenceDiagram`
- 节点色值：核心功能 #ccffcc，中性 #e1f5ff
- 每个图表下方添加明确的说明文字

## 保存位置

`docs/02_需求任务/{功能名称}.md`

## 质量检查清单

### P0 - 必须通过

- **用户故事表格**：完整提取自需求文档，包含优先级图标
- **主要操作场景**：每个P0用户故事至少有2个主要操作场景定义，每个P1用户故事至少有1个
- **Mermaid图表**：至少包含功能分解图、用户流程图、功能流程图
- **验收标准**：P0项清晰可测试
- **影响分析章节存在**：包含"影响分析"章节，且包含完整的引用清单表格
- **引用清单基于 Grep 实际搜索**：引用清单中每条记录的文件路径和行号均来自 Grep 实际搜索结果，**不得估算或凭记忆填写**
- **改动范围汇总存在**：包含直接修改文件数、间接影响文件数、未覆盖风险说明

### P1 - 应该通过

- **引用清单影响级别标注完整**：每条引用记录均有 🔴/🟡/🟢 级别标注及说明
- **时序图**：包含完整的时序交互图
- **使用场景**：3-5个场景，采用"背景-操作-结果"格式
- **图表说明**：每个Mermaid图表都有明确的说明
- **操作场景完整性**：主要操作场景包含前置条件、操作步骤、预期结果、验证关注点

### P2 - 可以有

- **功能详情**：每个功能点都有详细说明
- **设计文档关联**：主要操作场景明确关联到设计文档的对应章节
