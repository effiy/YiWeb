# 验证门禁规范

> 本规范约束 implement-code 技能在**阶段 3（验证门禁）**的决策逻辑与执行要求。  
> **核心意图**：没有通过门禁的项目，禁止进入代码实施阶段。门禁是不可跳过的强制关卡。

---

## 1. 门禁原则（P0 — 不可违反）

| 编号 | 原则 |
|------|------|
| G0-1 | P0 检查项有任意一项未通过，**禁止进入阶段 4**。没有例外。 |
| G0-2 | 门禁失败时，**必须** 修复测试页面（阶段 2），不得修改动态检查清单以"降级"P0 项。 |
| G0-3 | 自修复上限为 **2 轮**；第 3 次仍失败时，停止并输出阻断报告，等待人工介入。 |
| G0-4 | 门禁结果 **必须** 完整保存，供阶段 5 过程总结引用。 |
| G0-5 | "人工确认"类检查项 **不计入** P0 通过/失败判定，但必须在报告中列出。 |

---

## 2. 检查项分类与验证方式

### 2.1 P0 — 必须通过（自动验证）

| 检查类型 | 验证方式 | 通过条件 |
|---------|---------|---------|
| UI 元素存在 | `expect(locator).toBeVisible()` | 元素可见，无控制台 JS 错误 |
| UI 元素可操作 | `expect(locator).toBeEnabled()` | 元素未禁用、未被遮挡 |
| 操作步骤可执行 | `page.click/fill` 不抛出异常 | 操作完成，无超时 |
| 桩行为触发 | 操作后检查状态变更 | 对应结果区域可见 / 文本更新 |
| data-testid 覆盖 | 枚举检查清单操作项，确认 testid 存在 | 无遗漏 testid |

### 2.2 P1 — 建议通过（自动 + 人工）

| 检查类型 | 验证方式 |
|---------|---------|
| 错误状态展示 | 触发错误条件，确认 `error-msg` 可见 |
| 边界值处理 | 输入极端值，确认无 JS 错误 |
| 可访问性 | ARIA 属性存在且语义正确 |

### 2.3 P2 — 可选（人工）

| 检查类型 | 说明 |
|---------|------|
| 视觉一致性 | 与设计稿或使用文档截图对比 |
| 响应式布局 | 在不同视口宽度下的表现 |
| 动画/过渡效果 | 是否符合交互设计说明 |

## 3. 验证执行流程

```
对每个场景（按动态检查清单顺序）：
  1. 打开对应原型页面
  2. 执行 P0 Playwright 检查
  3. 记录结果（通过 / 未通过 / 人工确认）
  4. 未通过 → 记录：检查项描述 + 实际行为 + 预期行为
  5. 本轮结束后统计 P0 通过率
```

---

## 4. 修复循环约束

```
轮次 1：
  失败项 → 分析根因（元素缺失 / testid 错误 / 桩行为未触发）
         → 仅修复测试原型页面（阶段 2 产物）
         → 不修改检查清单内容
         → 重新运行阶段 3

轮次 2：
  失败项 → 同上
         → 若根因指向需求歧义（检查清单与设计文档矛盾），
           在报告中标注"文档冲突"，等待人工裁决后继续

轮次 3（上限）：
  输出阻断报告（见第 5 节），停止执行
  等待人工确认修复方向后，从阶段 2 重新开始
```

---

## 5. 阻断报告格式

```
=== 验证门禁阻断报告 ===
功能：<功能名>
执行轮次：3（已达上限）
时间：<ISO 时间戳>

P0 未通过项：
  场景：<场景名>
  检查项：<检查项描述>
  来源：动态检查清单 § <章节> / 需求任务 US-{N}
  实际行为：<Playwright 错误信息或截图描述>
  预期行为：<来自动态检查清单的预期结果>
  根因分类：□ UI 元素缺失  □ testid 错误  □ 桩行为未触发  □ 文档冲突  □ 其他
  建议操作：<具体建议>

人工确认项：
  - <项目描述>（需确认：<确认内容>）

结论：阶段 4（代码实施）已被阻断。
      请根据上述根因分析修复后，从阶段 2 重新开始。
```

---

## 6. 门禁通过报告格式

```
=== 验证门禁通过报告 ===
功能：<功能名>
执行轮次：<N>（共 <M> 个场景）
时间：<ISO 时间戳>

P0 总计：<N> 项，全部通过 ✅
P1 通过：<N1> 项 / 总 <M1> 项
P2 通过：<N2> 项 / 总 <M2> 项（仅供参考）

人工确认项（<K> 项，不计入门禁）：
  - <项目描述>

结论：验证门禁已通过，解锁阶段 4（项目代码实施）。
```

---

## 7. 禁止事项

- ❌ 将 P0 检查项降级为 P1 以"通过"门禁
- ❌ 在门禁未通过时直接跳到代码实施
- ❌ 通过修改动态检查清单的"预期结果"来让断言通过
- ❌ 将"人工确认"项标记为自动通过
- ❌ 修复轮次超过 2 次后仍继续尝试（必须停止并输出阻断报告）
