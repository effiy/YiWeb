# 项目报告规范

> **生成约束**：本文档类型**禁用模板**。即使 `templates/` 下存在同名文件，也不作为生成输入。
> 所有内容必须来自真实的代码变更（git diff / 文件对比），无真实来源则写"待补充"，**不得编造**。

## 文档概述

- 项目报告是功能交付后的变更存档文档，如实记录"改了什么、改前是什么、改后是什么"
- **上游文档**：设计文档（`docs/03_设计文档/`）
- **Grounding 来源**：`git diff`、实际源文件读取、构建日志

## 生成前置条件（P0 硬门槛）

以下任一未满足，**必须中止生成** 并向用户要求补齐：

1. 能够访问本次变更的文件列表（git diff --name-only 或用户提供）。
2. 每个变更文件的"变更前"内容可读取（git show HEAD~1:<path> 或原始内容已提供）。
3. `docs/03_设计文档/<功能名>.md` 存在（用于关联文档链接）。

## 文档结构（严格遵循）

### 1. 文档头部（必须包含）

```markdown
# {功能名称}项目报告

> **文档版本**: v1.0 | **最后更新**: {YYYY-MM-DD} | **维护者**: YiAi 团队
>
> **关联文档**: [需求文档](../01_需求文档/{文件名}.md) | [需求任务](../02_需求任务/{文件名}.md) | [设计文档](../03_设计文档/{文件名}.md) | [使用文档](../04_使用文档/{文件名}.md)
>

[变更概述](#变更概述) | [变更文件列表](#变更文件列表) | [变更前后对比](#变更前后对比) | [变更汇总表](#变更汇总表)

---
```

### 2. 变更概述

- 简要描述本次交付的目标与范围，50-100 字
- 列出核心变更数量（新增 N 文件 / 修改 N 文件 / 删除 N 文件）
- **来源**：取自 git commit message 或用户描述，不得凭空添加说明

### 3. 变更文件列表（必须，强制完整）

列出**所有**变更文件，不得遗漏、不得新增：

| 序号 | 文件路径 | 变更类型 | 说明 |
|------|---------|---------|------|
| 1 | `src/...` | 新增 / 修改 / 删除 | {一句话说明改动目的} |

**变更类型取值**：新增 / 修改 / 删除 / 重命名

**来源约束**：文件路径必须取自 git diff 或用户确认的清单；禁止根据设计文档"猜测"应该变更哪些文件。

### 4. 变更前后对比（必须，每个变更文件一节）

每节结构：

```markdown
#### {文件路径}

**变更类型**：{新增 / 修改 / 删除}

**变更前**：

```{语言}
{变更前的完整相关代码或"（新增文件，无变更前内容）"}
```

**变更后**：

```{语言}
{变更后的完整相关代码或"（已删除，无变更后内容）"}
```

**说明**：{一句话说明本次变更的原因与影响}
```

**约束**：
- "变更前"内容必须来自真实的旧版本，不得根据"变更后"反推
- 若变更文件行数超过 100 行，展示关键片段（函数/类级别），并注明"已截取关键段落，完整内容见文件"
- 新增文件："变更前"写 `（新增文件，无变更前内容）`
- 已删除文件："变更后"写 `（已删除，无变更后内容）`

### 5. 变更汇总表（必须）

| 文件路径 | 变更类型 | 影响评估 | 关键变更说明 |
|---------|---------|---------|------------|
| `src/...` | 修改 | 低 / 中 / 高 | {关键变更说明} |

**影响评估标准**：
- **高**：影响公共接口 / Store / 路由 / 构建配置
- **中**：影响单个功能模块的行为
- **低**：样式调整 / 注释 / 文档

## 保存位置

- 单独生成：`docs/05_项目报告/{功能名称}.md`
- 全文档生成：`docs/00_rYr/<功能名>/项目报告.md`

## 质量检查清单

### P0 - 必须通过

- **文档头部完整**：含版本、日期、维护者、关联文档
- **变更文件列表完整**：与 git diff 或用户提供的清单完全一致
- **每文件有前后对比**：变更文件列表中的每个文件都有对应的"变更前后对比"节
- **变更前内容真实**：来自旧版本，非推断
- **代码路径真实**：文件路径在仓库中真实存在
- **变更汇总表完整**：所有变更文件都出现在汇总表中

### P1 - 应该通过

- **变更说明清晰**：每条变更有明确的"原因与影响"说明
- **影响评估合理**：高/中/低评估与实际改动范围匹配
- **关键代码截取得当**：超长文件只展示关键段落，并注明截取

### P2 - 可以有

- **变更概述数据准确**：新增/修改/删除文件数与列表一致
- **导航链接存在**：头部有章节跳转链接
