# 需求任务文档规范

## 文档概述

- 需求任务文档是对需求文档中用户故事的细化和功能分解，用于指导开发实施
- 必须从对应需求文档提取用户故事表格，保持需求与任务的一致性
- 上游文档：需求文档（位于 `docs/01_需求文档/`）
- 下游文档：设计文档

## 文档结构（严格遵循）

### 1. 文档头部（必须包含）

```markdown
# {功能名称}

> **文档版本**: v1.0 | **最后更新**: {日期} | **维护者**: YiAi 团队
>
> **关联文档**: [需求文档](../01_需求文档/{文件名}.md) | [设计文档](../03_设计文档/{文件名}.md) | [使用文档](../04_使用文档/{文件名}.md)
>

[功能概述](#功能概述) | [功能分析](#功能分析) | [功能详情](#功能详情) | [验收标准](#验收标准) | [使用场景示例](#使用场景示例)

---
```

### 2. 功能概述

- 简要说明功能的目标和范围，100-200字
- 列出 3 个核心价值点，使用 🎯⚡📖 图标

### 3. 功能分析（必须包含 Mermaid 图表）

必须包含 4 种 Mermaid 图表：

- **功能分解图**：`graph TB` - 展示功能模块和子功能
- **用户流程图**：`flowchart TD` - 展示用户操作流程
- **功能流程图**：`flowchart TD` - 展示系统功能处理流程
- **完整时序图**：`sequenceDiagram` - 展示组件间交互时序

每个图表下方必须添加明确的说明文字。

### 4. 用户故事表格（必须从需求文档提取）

- 完整提取自需求文档的用户故事表格
- 保持优先级图标（🔴🟡🟢）和格式不变
- 包含过程生成文档和产出智能文档列
- **必须包含主要操作场景描述**：在用户故事中明确列出主要操作场景，用于后续检查清单的动态生成

### 5. 主要操作场景定义（必须包含）

为每个用户故事明确列出主要操作场景，格式如下：

```markdown
#### 🎯 主要操作场景：{场景名称}

**场景描述**：{简要描述这个操作场景}

**前置条件**：{列出执行此操作前需要满足的条件}

**操作步骤**：
1. {步骤1}
2. {步骤2}
3. {步骤3}

**预期结果**：{描述操作完成后的预期结果}

**验证关注点**：{列出验证此场景需要检查的关键点}

**相关设计文档章节**：{指向设计文档中对应的实现章节}
```

主要操作场景的要求：
- 每个 P0 用户故事至少包含 2 个主要操作场景
- 每个 P1 用户故事至少包含 1 个主要操作场景
- 场景应覆盖核心用户路径和关键功能点
- 场景描述应具体、可操作、可验证

### 6. 功能详情

- 按功能点分章节
- 每个功能点包含：功能说明、价值、解决的痛点、收益
- 复杂功能点可添加 Mermaid 图表

### 7. 验收标准

- **P0 - 必须通过**：核心功能，不实现无法发布
- **P1 - 应该通过**：重要功能，建议实现
- **P2 - 可以有**：锦上添花，可选实现

### 8. 使用场景示例

- 每个场景包含：背景、操作、结果
- 使用 📋🎨 等图标标识不同场景

## Mermaid 图表规范

- 功能分解用 `graph TB`
- 用户流程用 `flowchart TD`
- 功能流程用 `flowchart TD`
- 时序图用 `sequenceDiagram`
- 节点色值：核心功能 #ccffcc，中性 #e1f5ff
- 每个图表下方添加明确的说明文字

## 保存位置

`docs/02_需求任务/{功能名称}.md`

## 质量检查清单

### P0 - 必须通过

- **用户故事表格**：完整提取自需求文档，包含优先级图标
- **主要操作场景**：每个P0用户故事至少有2个主要操作场景定义，每个P1用户故事至少有1个
- **Mermaid图表**：至少包含功能分解图、用户流程图、功能流程图
- **验收标准**：P0项清晰可测试

### P1 - 应该通过

- **时序图**：包含完整的时序交互图
- **使用场景**：3-5个场景，采用"背景-操作-结果"格式
- **图表说明**：每个Mermaid图表都有明确的说明
- **操作场景完整性**：主要操作场景包含前置条件、操作步骤、预期结果、验证关注点

### P2 - 可以有

- **功能详情**：每个功能点都有详细说明
- **设计文档关联**：主要操作场景明确关联到设计文档的对应章节
